# Сетевая изоляция Docker-развертываний

При запуске Hermes RU Iola в Docker стандартный режим `network_mode: host`
дает процессу агента прямой доступ к сети хоста. Для публичного или
многопользовательского развертывания лучше разделить внутренние сервисы и
компоненты, которым нужен внешний доступ.

Основная цель такой схемы - снизить риск утечки данных через команды,
которые агент может выполнить в терминальном backend после prompt injection.

## Модель угроз

Модель доверия описана в [SECURITY.md](../../SECURITY.md). Терминальный
backend остается главным рубежом исполнения команд. Сетевая изоляция добавляет
второй уровень защиты: внутренние сервисы работают в сети без прямого выхода
наружу, а внешний доступ получает только тот сервис, которому он действительно
нужен.

## Архитектура

```
Docker network: internal
  hermes-agent
  hermes-dashboard
  hermes-gateway

Docker network: external
  hermes-gateway
```

Используются две Docker-сети:

- `internal` - внутренняя сеть без прямого выхода наружу. Здесь работают агент,
  dashboard и внутренний интерфейс gateway.
- `external` - сеть для компонентов, которым нужен доступ к API провайдеров,
  мессенджеров и другим внешним сервисам.

Gateway подключается к обеим сетям: принимает внешние события и передает их
агенту во внутреннем контуре.

## Пример override-конфигурации

Создайте `docker-compose.override.yml` рядом с основным `docker-compose.yml`:

```yaml
networks:
  internal:
    driver: bridge
    internal: true
  external:
    driver: bridge

services:
  gateway:
    network_mode: ""
    networks:
      - internal
      - external
    ports:
      - "127.0.0.1:9119:9119"

  dashboard:
    network_mode: ""
    networks:
      - internal
```

Запуск:

```bash
HERMES_UID=$(id -u) HERMES_GID=$(id -g) docker compose up -d
```

## Проверка

После запуска проверьте, что внутренние сервисы доступны, а компоненты без
внешней сети не могут обращаться наружу:

```bash
docker compose exec gateway \
  curl -sf --max-time 5 http://hermes-dashboard:9119/health && echo "OK" || echo "FAIL"
```

Если вы добавляете новые контейнеры, подключайте их к `external` только когда
им действительно нужен внешний доступ.

## Ограничения

- DNS-запросы могут обрабатываться отдельно от HTTP-доступа. Если нужна более
  строгая модель, настройте отдельный DNS-контур для Docker-сетей.
- Эта схема изолирует сетевой контур контейнеров, но не заменяет sandbox для
  исполнения команд. Для более жесткой изоляции используйте отдельный
  терминальный backend.
- Адаптеры мессенджеров и LLM-провайдеров требуют внешнего доступа. Держите
  такие интеграции в отдельном сервисе или сетевом контуре.
